# Dev Dashboard

## Product Vision

Dev Dashboard is a personal operating system for a developer.

It is not just a task tracker and not just a habit tracker.
It is a system that helps the user:

- track long-term goals
- maintain daily discipline
- review each day
- monitor personal projects
- understand personal progress over time

The product should feel like a serious personal tool, not a toy app.

The goal is to build a project that is:

1. genuinely useful for the owner on a daily basis
2. architecturally strong as a backend portfolio project
3. easy to extend later as part of a bigger personal ecosystem

---

## Core Product Idea

The dashboard should answer these questions:

- What are my current important goals?
- What habits did I complete today?
- Did I review my day?
- How am I progressing over time?
- Which personal projects are active right now?
- Am I staying disciplined or drifting?

This is a personal system for self-management, discipline, progress, and reflection.

---

## Product Philosophy

Build for extension, not for prediction.

Correct philosophy:

small useful core  
→ real daily usage  
→ better understanding of needs  
→ careful expansion

Do NOT attempt to build the ideal full product on day one.

---

## Frontend Philosophy

Use:

- Django templates
- HTMX
- minimal JavaScript only when necessary

Do NOT use React/Vue/heavy frontend tooling.

The frontend should remain server-rendered and dynamic.

Primary focus remains:

- domain design
- backend architecture
- maintainability
- usefulness

---

## Responsive / Mobile UX

The product must work comfortably on both desktop and mobile devices.

UI rules:

- prioritize fast daily interactions
- support narrow screens cleanly
- avoid desktop-only layouts
- optimize key actions for mobile:
  - viewing today's dashboard
  - logging habits
  - updating daily review
  - checking active goals

Design direction:

- desktop may use multi-column layout
- mobile should collapse into single-column layout
- forms should remain short and practical
- HTMX should reduce friction on mobile

Principle:

Desktop for depth, mobile for daily use.

---

## Architecture Principles

### Domain rules

- All write operations must go through services
- All read-heavy queries and aggregations must go through selectors
- Views must remain thin
- Forms must remain thin
- Templates must not contain business logic
- Multi-step write workflows should use `transaction.atomic()` when needed
- Avoid fat models
- Avoid duplicated validation
- Avoid premature abstractions

### Extensibility rules

- Keep domain modules loosely coupled
- Design for future expansion without rewriting core
- Do not tightly couple future ecosystem features into MVP

Long-term and medium-term goals.

### habits
Habits and daily tracking.

### reviews
Daily reflections / reviews.

### dashboard
Aggregated homepage/dashboard logic.

### projects_overview
High-level project tracking.

### api
Optional DRF layer for future integrations.

### core
Shared utilities only:
- helpers
- shared exceptions
- mixins
- utility functions

Use HTMX selectively.

- habit logging
- review updates
- inline status updates
- widget refreshes

---

## Services / Selectors Pattern

### Services

All writes go through app-level services.py.

Examples:

- create_goal()
- complete_goal()
- log_habit_entry()
- create_or_update_review()

### Selectors

All reads go through selectors.py.

Examples:

- get_active_goals()
- get_today_habits()
- get_dashboard_context()

---

## Testing Strategy

Prioritize:

1. service tests
2. selector tests
3. major flow tests

Critical cases:

- goal lifecycle
- habit logging
- duplicate prevention
- review uniqueness
- dashboard aggregation

Avoid excessive cosmetic HTML tests.

---

## Database Strategy

### Development

Use SQLite by default.

### Production

Use PostgreSQL.

### Rules

- If DATABASE_URL exists → use PostgreSQL
- Otherwise → fallback to SQLite

Avoid PostgreSQL-only features during MVP.

Examples to avoid early:

- ArrayField
- PostgreSQL-only JSON querying
- PostgreSQL extensions

Maintain database-agnostic architecture where practical.
